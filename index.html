🛡️ Security Architecture
How M-Pay protects you without internet
📌 Daily Limit Lock (Beta)ACTIVE
₹1,000/day hard cap enforced on-device. Limit tied to bank-issued offline token. Resets at midnight IST. Cannot be overridden offline.
🔐 PIN → AES-256 EncryptionACTIVE
PIN is hashed with PBKDF2 (100,000 iterations + salt). The resulting key encrypts your entire wallet using AES-256-GCM. PIN never stored — only the derived key.
🔑 Bank-Issued Identity CertVALID
Your bank issues an X.509-style offline certificate during setup. Every payment carries this cert. Recipients verify it against the bank's public key (hardcoded in app) — no server call needed.
🎫 Single-Use Spending TokensACTIVE
Each payment consumes one token (serial number). Tokens cannot be reused — double-spend attempts are rejected immediately. Token chain verified by both devices.
✍️ HMAC-SHA256 SignaturesACTIVE
Every transaction payload is signed with HMAC-SHA256 using your device key. Recipient verifies signature before accepting. Tampered packets are rejected.
⏱️ 96h Expiry + Replay GuardACTIVE
Signed transactions include a timestamp + nonce. Packets older than 96h or with a replayed nonce are auto-rejected. Prevents replay attacks even offline.
🚫 Brute-Force LockoutACTIVE
5 failed PIN attempts triggers a 30-second lockout. After 3 lockouts, wallet data is wiped and requires bank re-setup. Attempt counter persisted to localStorage.
⏰ Auto-Lock (5 min idle)ACTIVE
Session expires after 5 minutes of inactivity. Wallet locks automatically — PIN required to re-enter. Timer resets on any user action.
🔍 Mutual AuthenticationACTIVE
Both devices verify each other's bank certificates before a payment begins. Neither device sends money to an unverified stranger — cert must be present and valid.
